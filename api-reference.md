## Overview

| Property | Value |
|----------|-------|
| **API Name** | Parcel App — External Deliveries API |
| **Base URL** | `https://api.parcel.app` |
| **Style** | REST (single read-only endpoint) |
| **Purpose** | Allows premium users to view upcoming and active deliveries on external devices and platforms (e.g., Home Assistant) |
| **Docs** | [parcelapp.net/help/api-view-deliveries.html](https://parcelapp.net/help/api-view-deliveries.html) |

> ⚠️ **Important**: This API returns cached responses from the app server. Calling the endpoint does **not** trigger a delivery update. It is not a full replacement for the Parcel app.

## Authentication

| Property | Value |
|----------|-------|
| **Type** | API Key |
| **Header Name** | `api-key` |
| **Key Generation** | [web.parcelapp.net](https://web.parcelapp.net) |

### Notes
- The API key is passed as a **custom HTTP header** (`api-key`), not via the standard `Authorization` header.
- No OAuth, bearer tokens, or token refresh mechanisms.
- No documented key rotation or expiration policy.

## Rate Limiting

| Property | Value |
|----------|-------|
| **Limit** | 20 requests per hour |
| **Window** | Rolling hourly |
| **Caching** | All responses are server-cached |

### Recommendations
- Poll no more frequently than every **3 minutes** to stay within limits.
- Since responses are cached, frequent polling yields no benefit.
- No documented rate-limit response headers (e.g., `X-RateLimit-Remaining`).

## Endpoints

### `GET /external/deliveries/`

Retrieve a list of recent or active deliveries for the authenticated user.

**Full URL**: `https://api.parcel.app/external/deliveries/`

#### Query Parameters

| Parameter | Type | Required | Default | Allowed Values | Description |
|-----------|------|----------|---------|----------------|-------------|
| `filter_mode` | `string` | No | `"recent"` | `"active"`, `"recent"` | Filter deliveries by their current status grouping |

- **`recent`**: Returns recently updated deliveries (including completed)
- **`active`**: Returns only deliveries currently in progress

#### Example Request

```bash
curl "https://api.parcel.app/external/deliveries/?filter_mode=active" \
  -H "api-key: YOUR_API_KEY"
```

#### Success Response (`200 OK`)

```json
{
  "success": true,
  "deliveries": [
    {
      "carrier_code": "ups",
      "description": "New laptop from Amazon",
      "status_code": 2,
      "tracking_number": "1Z999AA10123456784",
      "events": [
        {
          "event": "Package arrived at facility",
          "date": "2025-01-15 14:30",
          "location": "London, UK",
          "additional": "Sorted for next-day delivery"
        }
      ],
      "extra_information": "SW1A 1AA",
      "date_expected": "2025-01-16 12:00",
      "date_expected_end": "2025-01-16 18:00",
      "timestamp_expected": 1736942400,
      "timestamp_expected_end": 1736964000
    }
  ]
}
```

#### Error Response

```json
{
  "success": false,
  "error_message": "Invalid API key"
}
```

## Data Models

### Top-Level Response

| Field | Type | Always Present | Description |
|-------|------|:--------------:|-------------|
| `success` | `boolean` | ✅ | Whether the request was successful |
| `error_message` | `string` | ❌ | Error description (only present on failure) |
| `deliveries` | `Delivery[]` | ❌ | Array of delivery objects (only present on success) |

---

### `Delivery` Object

| Field | Type | Always Present | Description |
|-------|------|:--------------:|-------------|
| `carrier_code` | `string` | ✅ | Internal carrier identifier. See [Supported Carriers](#supported-carriers) |
| `description` | `string` | ✅ | User-provided label created when the delivery was added |
| `status_code` | `integer` | ✅ | Current delivery status. See [Status Codes](#delivery-status-codes) |
| `tracking_number` | `string` | ✅ | Carrier tracking number |
| `events` | `Event[]` | ✅ | Chronological delivery events (may be empty `[]`) |
| `extra_information` | `string` | ❌ | Additional tracking info (postcode, email, etc.) required by carrier |
| `date_expected` | `string` | ❌ | Expected delivery date/time. **No timezone information** |
| `date_expected_end` | `string` | ❌ | End of delivery window (indicates carrier provides a time range) |
| `timestamp_expected` | `integer` | ❌ | Unix epoch for expected delivery. Only available when carrier provides full date/time/timezone |
| `timestamp_expected_end` | `integer` | ❌ | Unix epoch for end of delivery window |

#### Date/Time Field Notes
- `date_expected` and `date_expected_end` are **timezone-naive strings** — display as-is.
- `timestamp_expected` and `timestamp_expected_end` are **Unix epoch integers** with full timezone context — use these for precise calculations when available.
- When both `date_expected` and `date_expected_end` are present, the carrier provides a **delivery window** (e.g., "between 12:00 and 18:00").

---

### `Event` Object

| Field | Type | Always Present | Description |
|-------|------|:--------------:|-------------|
| `event` | `string` | ✅ | Description of the delivery event |
| `date` | `string` | ✅ | Date/time of the event |
| `location` | `string` | ❌ | Location where the event occurred |
| `additional` | `string` | ❌ | Supplementary carrier information |

## Delivery Status Codes

| Code | Status | Description | Action Required? |
|------|--------|-------------|:----------------:|
| `0` | **Completed** | Delivery has been completed | No |
| `1` | **Frozen** | No updates for extended period; unlikely to update further | ⚠️ Investigate |
| `2` | **In Transit** | Package is in transit | No |
| `3` | **Awaiting Pickup** | Delivery awaiting recipient pickup | ✅ Yes |
| `4` | **Out for Delivery** | Package is out for delivery | No |
| `5` | **Not Found** | Carrier cannot find the delivery | ⚠️ Investigate |
| `6` | **Failed Attempt** | Delivery attempt was unsuccessful | ✅ Yes |
| `7` | **Exception** | Something is wrong; requires attention | ✅ Yes |
| `8` | **Info Received** | Carrier has shipment info but hasn't physically received the package yet | No |

### Status Grouping

| Category | Codes | Description |
|----------|-------|-------------|
| **Terminal** | `0`, `1` | No further updates expected |
| **Active** | `2`, `3`, `4`, `8` | Delivery is progressing normally |
| **Attention** | `5`, `6`, `7` | May require user intervention |

A complete list of supported carrier codes is available at:

## Error Handling

| Scenario | Response |
|----------|----------|
| Invalid or missing API key | `{ "success": false, "error_message": "..." }` |
| Rate limit exceeded | `{ "success": false, "error_message": "..." }` |
| Invalid `filter_mode` value | Behavior undocumented — recommend defaulting to valid values |

### Notes
- The API uses **response-body error signaling** via the `success` boolean rather than (or in addition to) HTTP status codes.
- No documented HTTP status code specifics (e.g., whether invalid auth returns `401` vs `403`).
- Always check `success` field first before processing `deliveries`.

## Integration Considerations

### For Agent C Tool Development
- **Single endpoint** — tool can expose one function with an optional `filter_mode` parameter.
- **API key configuration** — should be stored as a workspace/environment secret, injected via header.
- **Rate limit management** — implement client-side throttling (≤20 req/hr). Consider caching responses locally.
- **Response parsing** — always check `success` before accessing `deliveries`.
- **Status code mapping** — provide human-readable status labels, not just integer codes.
- **Carrier code enrichment** — optionally fetch and cache the supported carriers list for display names.

### Suggested Tool Functions

| Function | Description | Parameters |
|----------|-------------|------------|
| `get_deliveries` | Fetch deliveries with optional filter | `filter_mode?: "active" \| "recent"` |
| `get_supported_carriers` | Fetch carrier code → name mapping | _(none)_ |
